# Core Overview

Status: partially locked

## Locked decisions

### Product shape

We are designing a generic personal/professional assistant platform first, then adding profession-specific packs on top.

Initial implementation focuses only on the Doctor profession pack.

### Profession selection

Profession selection is manual only for now.

The user explicitly chooses the active mode, e.g.

```text
Doctor Mode
Founder Mode
Developer Mode
Lawyer Mode
```

No automatic profession detection in the MVP.

### Transcript input

The primary input contract is streaming transcript chunks.

The system should be able to receive partial transcript updates over time, not only a final completed transcript.

Example conceptual stream:

```json
{
  "session_id": "session_001",
  "chunk_id": "chunk_014",
  "speaker": "Person 1",
  "text": "Do you have any chest pain or difficulty breathing?",
  "timestamp": "2026-06-20T10:30:14Z"
}
```

### Output shape

Outputs should be custom per profession.

We are not enforcing one universal artifact schema across all professions. The core platform will provide generic orchestration, but each profession pack can define its own output shape.

Example:

```text
Doctor Mode → SOAP note, ICD suggestions, care plan, patient memory
Developer Mode → GitHub issue, acceptance criteria, PR draft
Founder Mode → CRM update, follow-up email, investor note
Lawyer Mode → matter note, billable entry, case summary
```

### Privacy/compliance layer

Privacy/compliance behavior should be toggleable on/off.

This allows different deployment modes:

```text
privacy_filtering: enabled
privacy_filtering: disabled
```

For Doctor Mode, privacy filtering will likely be enabled in the intended flow, but the platform should not hardcode that globally.

### Memory

Memory is profession-specific.

For now, the LLM/profession agent can decide what should be stored, ignored, updated, or proposed as memory.

Memory updates should be generated by the active profession pack, not by a generic global memory schema.

### Action execution

Action execution should be toggleable on/off.

When action execution is off, the system should only produce proposed actions.

When action execution is on, the system may route approved or allowed actions to integrations such as Notion, Gmail, Calendar, Sheets, Slack, GitHub, or future domain-specific tools.

### First profession pack

The first profession pack is Doctor Mode.

### Demo scope

Initial demo focuses only on Doctor Mode.

Other professions can be mentioned as future packs, but the first implementation should go deep on Doctor Mode.

## Open decisions

### Product name

Locked: **Carry**

Rationale: Carry suggests carrying context forward from conversations into future work, memory, follow-ups, and tools. It is simple, broad, profession-agnostic, and friendly without sounding like a generic chatbot.

### One-line platform statement

Locked draft:

```text
Carry is a voice-first assistant that carries context from professional conversations into notes, memory, follow-ups, and work actions.
```

Supporting tagline options:

```text
Carry turns conversations into context-aware work.
Carry remembers what matters and moves work forward.
Carry brings the context of every conversation into your next action.
```
